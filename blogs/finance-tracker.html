Executive Summary

In the current FinTech landscape, users often trade data privacy for convenience. Commercial Personal Finance Management (PFM) apps rely on intrusive bank account scraping, impose rigid categorization taxonomies, and create data silos where investment tracking and expense management rarely intersect.

To address these constraints, I architected a custom Financial Data Ingestion & Analytics Pipeline. Built on a serverless Google Cloud stack, this system decouples data capture from storage, utilizing an event-driven ETL middleware to normalize transactions in real-time. The result is a zero-latency, privacy-centric ledger that supports granular asset allocation modeling.

The Business Challenge

The objective was to solve three critical friction points inherent in commercial FinTech products:

  • Latency & Reliability: Third-party aggregators often suffer from broken bank syncs and OTP failures, leading to "stale" financial data.
  • Taxonomy Rigidity: Standard apps force users into generic categories (e.g., "Food," "Travel") that fail to distinguish between CapEx vs. OpEx in a personal context.
  • Data Sovereignty: Financial data is a sensitive asset. Handing it to "free" apps introduces privacy risks and vendor lock-in.

Engineering Goal

Build a system that offers the speed of a mobile app, the flexibility of a relational database, and the privacy of a local ledger.

System Architecture

The system follows a standard ETL (Extract, Transform, Load) pattern, leveraging the Google Workspace ecosystem as a serverless backend.

Ingestion Layer (Extract)

To minimize friction, I deployed two distinct Mobile-First Ingestion Interfaces using Google Forms:

  • Investment Ingestion: Captures asset class (Equity, Debt, Gold), ticker symbol, and capital deployed.
  • Expense Ingestion: Captures transaction value, category (Need/Want), and payment mode.

Middleware Layer (Transform)

Raw data is intercepted by an Event-Driven Trigger running on Google Apps Script. This middleware performs three critical functions:

  • Sanitization: Validates numeric inputs and standardizes timestamp formats (ISO 8601).
  • Normalization: Maps raw category inputs to a standardized backend taxonomy.
  • Routing: Dynamically routes the data payload to specific monthly shards to prevent database bloat.

Storage & Analytics Layer (Load)

The Master Ledger acts as a relational database where "Investments" and "Expenses" are stored in normalized tables. A separate Analytics Engine queries this data to generate a Real-Time Dashboard.

Implementation Logic (The "Secret Sauce")

The core of the system is the Dynamic Linking Algorithm that connects the "Spending" module with the "Investing" module—a feature missing in most commercial apps.

Automated Asset Allocation Tracking

Unlike standard trackers, this engine treats "Investments" as a distinct outflow category that maps to a Net Worth Asset Class.

Logic Example: When ₹5,000 is logged as "SIP Investment":

  1. The middleware tags it as an "Expense" for cash-flow purposes (money left the bank).
  2. Simultaneously, it creates a "Credit" entry in the Asset Register (Asset Value increased).
  3. This ensures the Savings Rate metric reflects true wealth accumulation.

Data Visualization & Analytics

The frontend dashboard provides a comprehensive view of financial health, updated instantly upon form submission.

Key Metrics Monitored:

  • Monthly Burn Rate: Operational expenditure vs. budget caps.
  • True Savings Rate: (Investments + Cash Retained) / Total Income.
  • Asset Allocation Delta: Current Portfolio split (e.g., 60% Equity / 40% Debt) vs. Target Allocation.

Future Roadmap

To scale this system beyond simple tracking into Predictive Analytics, I am currently developing a Python microservice using the gspread and pandas libraries.

  • Monte Carlo Simulations: Forecasting portfolio growth based on historical volatility.
  • XIRR Calculation: Automating complex return calculations.
  • Anomaly Detection: Using scikit-learn to flag spending outliers.

Conclusion

This project demonstrates that robust financial engineering does not require enterprise-grade budgets. By leveraging low-code tools with custom scripting and strict data discipline, I built a system that offers 100% data sovereignty, zero maintenance costs, and infinite customizability.
